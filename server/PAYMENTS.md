# GCash payments with PayMongo

## Flow

1. The customer chooses GCash during checkout.
2. The API creates the order in `PAYMENT_PENDING`, reserves its stock, and creates a PayMongo Hosted Checkout Session.
3. Only the hosted `checkout_url` is returned to the browser. The PayMongo secret key never leaves the API server.
4. PayMongo redirects the customer to `/payment/return?order=...`.
5. The return page polls the authenticated payment-status endpoint for display only.
6. The signed `checkout_session.payment.paid` webhook is the source of truth. The API checks its signature, event ID, order reference, amount, and PHP currency before marking the payment `PAID` and the order `CONFIRMED`.

`PaymentEvent.providerEventId` is unique, so webhook retries cannot process an event twice. A failed or expired event cancels a still-pending order and releases reserved inventory.

If PayMongo checkout creation fails, the API cancels the new order, releases inventory, and restores the selected products to the cart.

## Database records

The existing Prisma schema already contains the required records:

- `Order`: internal order, fulfillment status, total, customer snapshot.
- `Payment`: gateway session ID in `providerReference`, hosted URL, amount, method, and payment status.
- `PaymentEvent`: immutable webhook audit record and idempotency key.
- `OrderStatusEvent`: customer/admin order timeline.
- `InventoryMovement`: reservation, sale, and release audit trail.

No personal GCash number or wallet credential is stored.

## Local test configuration

Copy the payment variables from `.env.example` into `.env`:

```env
GCASH_ENABLED="true"
PAYMENT_PROVIDER="paymongo"
PAYMONGO_SECRET_KEY="sk_test_your_test_secret_key"
PAYMONGO_WEBHOOK_SECRET="your_endpoint_secret"
PAYMONGO_API_URL="https://api.paymongo.com/v2"
PAYMENT_SUCCESS_URL="http://localhost:5173/payment/return"
PAYMENT_CANCEL_URL="http://localhost:5173/cart"
```

GCash is disabled by default. Keep `GCASH_ENABLED="false"` on the backend and
omit `VITE_GCASH_ENABLED` (or set it to `"false"`) on the frontend until the
merchant account is ready. To enable it later, set both flags to `"true"` and
restart the backend and frontend.

Use PayMongo test keys locally. A webhook cannot call `localhost`; expose the backend with an HTTPS development tunnel and register:

```text
https://your-public-test-host/api/webhooks/payments/paymongo
```

Subscribe that endpoint to `checkout_session.payment.paid`. Copy the endpoint secret generated by PayMongo into `PAYMONGO_WEBHOOK_SECRET`, then restart the backend.

## Production checklist

- Complete PayMongo business verification and enable GCash for the merchant account.
- Replace `sk_test_...` with the live secret in the deployment's encrypted environment settings.
- Set `APP_ORIGIN`, `PAYMENT_SUCCESS_URL`, and `PAYMENT_CANCEL_URL` to the production HTTPS frontend.
- Register the production HTTPS webhook separately and store its production endpoint secret.
- Keep `ENFORCE_HTTPS=true`.
- Never place `PAYMONGO_SECRET_KEY` or `PAYMONGO_WEBHOOK_SECRET` in Vite variables, frontend code, source control, screenshots, or logs.
- Run a low-value live end-to-end payment before opening checkout to customers.
